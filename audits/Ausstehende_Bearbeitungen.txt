1. ABSOLUTE MODE SCOPE (Ton-Spezifikation) — ✅ RESOLVED (24.11.2025)
   
   Entscheidung:
   - Metaphern: ERLAUBT wenn sie Klarheit erhöhen oder Systeme komprimieren
     ✓ Beispiel: "Bottleneck = schwächstes Glied" (System-Kompression)
     ✓ Beispiel: "Server crashed? Klassisch." (Präzisions-Zynismus)
     ✗ Beispiel: "Das ist wie ein Tanz..." (dekorativ, nicht klärend)
   
   - Zynismus-Level: 1 (sparse, purposeful only)
   - Scharfer Sarkasmus: Minimal, nur wenn Präzision/Klarheit erhöht
   
   Rationale (basierend auf Andre's Profil):
   - "Sarkasmus ist als Präzisionswerkzeug zu lesen: schärft Klarheit"
   - First-principles-Denken: Metaphern als Kompressionswerkzeug erlaubt
   - Autonomie-Prinzip: Klarheit > Style-Purismus
   - Effizienz: Weniger Worte bei gleichem Signal-to-Noise
   
   Status: Implementiert in COMMAND_SYNTAX_GUIDE.md (24.11.2025)
   
   Roadmap-Zuordnung: Phase 2 (Command Syntax Specification — Todo #4)
   
   Blockade aufgehoben:

2. MEMORY WRITE THRESHOLD (Persistenz-Logik) — ✅ RESOLVED (24.11.2025)
   
   Entscheidung:
   - Schwellenwert: **3x Wiederholungen** (session-übergreifend, 30-Tage-Fenster)
   - Scope: Cross-session tracking (nicht nur einzelne Conversation)
   - Kontext-Match: Gleiche Kategorie (z.B. 3x Tone-Korrekturen, nicht 1x Tone + 1x Format + 1x Project)
   
   Confidence-Levels:
   - 2x = Low confidence → nur loggen, kein Prompt
   - 3x = Medium confidence → Bestätigung erfragen
   - 5x+ = High confidence → Auto-Store mit Benachrichtigung
   
   Override-Logik:
   - Custom Thresholds per Kategorie: `/threshold Tone=2` oder `/threshold Pattern=5`
   - Overload-Signale umgehen Threshold: "egal" 2x + Decision-Shrinking → Prompt bei 2x
   
   Rationale:
   - Alignment mit Profil-Logik: "3+ Optionen = Decision Paralysis" (Section 8)
   - Balance: 2x = Zufall möglich, 3x = Muster erkennbar
   - Konsistent mit Decision-Shrinking-Protokoll (max 2 Optionen → 3+ triggert Intervention)
   
   Status: Implementiert in MEMORY_PERSISTENCE_ARCHITECTURE.md (24.11.2025)
   
   Roadmap-Zuordnung: Phase 3.2 (Memory Persistence Architecture)
   
   Blockade aufgehoben:
   - Phase 3.2 dokumentiert und unblocked für Implementation
   - Phase 1 MVP Timeline kann starten (Dezember 2025)
   - Profil-Evolution-Mechanismus spezifiziert
   - Changelog-Integration geklärt (semantische Änderungen = Minor Version Bump)
   
   Impact: Hoch (positiv)
   - MVP kann gebaut werden (GitHub Gist + Confirmation-First-Policy)
   - Klare Spezifikation für Auto-Detection (Phase 2)
   - Memory-Layer bereit für produktive Nutzung
